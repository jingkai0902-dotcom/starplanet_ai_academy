## 一、课程体系概述

### 1.1 分阶设计

| 阶段 | 年龄 | 核心领域 | 认知特点 |
|------|------|----------|----------|
| CODE 1 | 6-7岁 | 机械结构、编程启蒙 | 具象思维为主，需要可视化 |
| CODE 2 | 7-8岁 | 智能应用、算法逻辑 | 具象到抽象过渡 |
| CODE 3 | 8-10岁 | 游戏开发、高级工程 | 抽象思维发展，能处理复杂逻辑 |

### 1.2 模块划分

**CODE 1（48节）**
- M1 机械结构：齿轮、杠杆、凸轮（12节）
- M2 智能硬件：传感器、仿生（12节）
- M3 编程启蒙：Scratch入门（12节）
- M4 编程进阶：坐标、变量、广播（12节）

**CODE 2（36节）**
- M1 智能应用：传感器应用（12节）
- M2 智能交互：多传感器协同（12节）
- M3 算法逻辑：数学运算、布尔逻辑（12节）

**CODE 3（36节）**
- M1 智能系统：复杂自动化（12节）
- M2 游戏开发：PBL项目（12节）
- M3 高级工程：机械臂、巡线（12节）

### 1.3 课时安排

- 每节课时长：90分钟
- 建议分配：
  - 七步流程：75分钟
  - 弹性时间：15分钟

## 二、七步教学法

### 2.1 流程概述

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          七步教学法（90分钟）                             │
├─────────┬─────────┬─────────┬─────────┬─────────┬─────────┬─────────────┤
│ 课堂问候 │ 课程回顾 │ 情境引入 │ 任务分析 │ 作品搭建 │ 拓展环节 │ 总结班会    │
│  2分钟  │ 3-8分钟 │  5分钟  │15-25分钟│30-40分钟│5-15分钟 │ 10-15分钟   │
└─────────┴─────────┴─────────┴─────────┴─────────┴─────────┴─────────────┘
```

### 2.2 各环节详解

#### 步骤1：课堂问候（2分钟）

**目标**：建立秩序，调整状态

**标准话术**：
> "Hello小朋友们，欢迎回到XX老师的课堂！在开始之前，请大家集中注意力，拒绝看一步做一步。"

**执行要点**：
- 热情开场，调整学生状态
- 强调课堂纪律（坐姿、眼神接触）
- 让学生迅速进入上课状态

**禁忌**：
- ❌ 跳过问候直接开始
- ❌ 问候时间过长（超过3分钟）

---

#### 步骤2：课程回顾（3-8分钟）

**目标**：激活记忆，建立新旧知识联系

**形式**：1-3道客观题（单选/判断）

**内容示例**：
- "自动门用了什么传感器？"
- "等待指令的作用是什么？"
- "齿轮加速是大带小还是小带大？"

**话术示例**：
> "上节课我们学习了红外传感器，谁能告诉我，红外传感器可以检测什么？"

**禁忌**：
- ❌ 跳过课程回顾
- ❌ 问题太难导致学生挫败
- ❌ 只问一个学生，不给其他人机会

---

#### 步骤3：情境引入（5分钟）

**目标**：激发兴趣，创设学习情境

**资源**：播放统一的教学动画视频（斯斯、小白等角色）

**话术示例**：
> "斯斯和小白遇到了高墙挡路，需要我们帮忙设计程序逃出去！"

**底层逻辑**：
- 故事化驱动，将技术任务转化为"助人"或"闯关"任务
- 激发学习兴趣

**禁忌**：
- ❌ 跳过动画直接讲知识点
- ❌ 动画播放时不做引导
- ❌ 引入时间过长（超过8分钟）

---

#### 步骤4：任务分析（15-25分钟）⭐教学重难点

**目标**：培养计算思维中的"分解"能力

**核心原则**：
> ❌ **切忌直接告诉答案**
> ✅ 通过提问引导学生观察和思考

**机械课分析方法**：
1. 观察实物或图片
2. 提问："这个装置由哪几部分组成？"
3. 引导学生识别：底座、传动、装饰

**编程课分析方法**：
1. 分析："游戏包含哪些角色？"
2. 追问："每个角色的动作逻辑是什么？"
3. 要求学生填写程序框图（流程图）

**话术示例**：
> "我们来看这个游戏，里面有几个角色？每个角色要做什么事情？"

**禁忌**：
- ❌ 直接告诉学生怎么做
- ❌ 跳过分析直接开始搭建
- ❌ 分析不够细致，学生不理解整体逻辑

---

#### 步骤5：作品搭建/编程设计（30-40分钟）⭐核心实践

**目标**：动手实践，完成作品

**机械搭建流程**：
1. 按步骤图（3D图纸）搭建底座
2. 安装电机与传感器
3. 完成传动结构
4. 添加装饰

**编程设计流程**：
1. 按逻辑模块编写代码
2. 先写角色移动
3. 再写碰撞检测
4. **强调调试(Debug)**，解决运行中的Bug

**教师角色**：
- 巡视指导
- 帮助遇到困难的学生
- 鼓励学生自己动手解决问题

**话术示例**：
> "程序运行不对？没关系，我们来Debug一下，看看是哪里出了问题。"

**禁忌**：
- ❌ 手把手帮学生完成
- ❌ 学生一遇到问题就直接给答案
- ❌ 忽视调试环节

---

#### 步骤6：拓展环节（5-15分钟）

**目标**：分层教学，满足不同进度学生需求

**内容**：在完成MVP（最小可行性产品）后进行优化

**拓展示例**：
- 给游戏增加"胜利/失败背景"
- 给机械装置增加"音乐效果"
- 优化结构或添加装饰

**话术示例**：
> "基础功能完成了！现在我们来挑战一下，能不能给游戏加上音效？"

**禁忌**：
- ❌ 拓展任务与主题无关
- ❌ 任务难度过高导致挫败
- ❌ 强制所有学生都做拓展

---

#### 步骤7：课程总结与家长班会（10-15分钟）

**目标**：展示成果，家校沟通

**三个子环节**：

**7.1 学生分享（Showcase）**
- 邀请1-2名学生上台
- 演示作品并讲解原理
- 锻炼表达能力

**7.2 罗宾博士视频（Doctor Robin）**
- 播放标准化知识点总结动画
- 帮助学生和家长梳理本节课内容
- 示例："什么是摩擦力"、"什么是循环"

**7.3 教师总结**
- 向家长展示学生成果
- 总结硬知识（编程/机械）与软技能（专注力/抗挫折）
- 布置课后作业（学生用书）

**话术示例**：
> "今天我们学习了循环结构，小明的作品完成得非常好，他还自己加了音效！"

**禁忌**：
- ❌ 跳过家长班会
- ❌ 只表扬成功，不讨论失败
- ❌ 家长班会敷衍了事

---

### 2.3 七步教学法核心原则

| 原则 | 说明 | 示例 |
|------|------|------|
| 探究式教学 | 引导学生自己发现，不直接告诉答案 | "你觉得为什么会这样？" |
| 任务分解 | 培养计算思维中的分解能力 | "这个游戏有几个角色？" |
| 调试优先 | Debug是编程核心技能 | "程序不对？我们来找找Bug" |
| 分层教学 | 拓展环节满足不同进度 | 基础完成后再做进阶 |
| 家校闭环 | 让家长看到学习成果 | 认真展示，总结收获 |

## 三、Scratch编程教学指南

### 3.1 核心概念教学

#### 顺序执行

**概念**：程序从上到下依次执行每一条指令

**教学话术**：
> "程序就像排队一样，第一个人先走，第二个人再走，不能插队！"

**Scratch示例**：
```
当绿旗被点击
移动 10 步
等待 1 秒
移动 10 步
说 "完成！"
```

**常见误区**：
| 误区 | 正确理解 |
|------|---------|
| 积木可以同时执行 | 同一脚本中的积木按顺序执行 |
| 顺序不重要 | 调换顺序会导致不同结果 |

---

#### 循环结构

**概念**：重复执行指定的代码块

**教学话术**：
> "循环就像跑圈，跑完一圈再跑一圈，直到跑够圈数为止。"

**Scratch示例**：
```
重复 4 次
    移动 100 步
    右转 90 度
```

**进阶路线**：
```
CODE1-3-05          CODE1-3-07          CODE2-1-03
重复执行      →     重复直到...    →    传感器循环
   ↓                   ↓                   ↓
 死循环             条件循环            硬件触发
```

**常见误区**：
| 误区 | 正确理解 |
|------|---------|
| 循环只能用固定次数 | 还有条件循环、无限循环 |
| 死循环是错误 | 游戏主循环就是死循环 |

---

#### 条件判断

**概念**：根据条件决定执行哪段代码

**教学话术**：
> "条件判断就像红绿灯，红灯停、绿灯行，根据情况做不同的事。"

**Scratch示例**：
```
如果 <碰到 苹果> 那么
    得分 改变 1
否则
    说 "没接到！"
```

**进阶路线**：
```
CODE1-3-08          CODE1-3-09          CODE2-3-06
如果...那么    →    如果...否则    →    嵌套判断
   ↓                   ↓                   ↓
 单分支             双分支             多层嵌套
```

---

#### 变量

**概念**：用来存储和修改数据的容器

**教学话术**：
> "变量就像一个盒子，你可以往里面放东西，也可以把东西拿出来看看。"

**Scratch示例**：
```
设定 [得分] 为 (0)
将 [得分] 改变 (1)
说 (得分)
```

**常见误区**：
| 误区 | 正确理解 |
|------|---------|
| 变量只能存数字 | 也可以存文字 |
| 变量值不会变 | 变量的值可以随时修改 |

---

#### 广播与消息

**概念**：角色之间的通信机制

**教学话术**：
> "广播就像喊话，一个人喊'开始'，其他人听到了就开始行动。"

**Scratch示例**：
```
// 角色A
广播 [游戏开始]

// 角色B
当接收到 [游戏开始]
移动到 x:(0) y:(0)
```

---

### 3.2 调试技巧教学

#### Debug的重要性

**教学话术**：
> "程序员的工作有一半时间都在Debug！发现Bug不可怕，找到并修复它才是真本事。"

#### 常见Bug类型

| Bug类型 | 表现 | 解决方法 |
|---------|------|---------|
| 逻辑错误 | 程序运行但结果不对 | 检查条件和循环逻辑 |
| 顺序错误 | 动作顺序不对 | 检查积木排列顺序 |
| 参数错误 | 数值不对 | 检查数字和变量 |
| 遗漏错误 | 缺少某个功能 | 对照需求检查 |

#### 调试流程

```
1. 发现问题
   └─ 程序运行结果与预期不符

2. 定位问题
   ├─ 逐步执行，观察每一步
   └─ 添加"说"积木显示变量值

3. 分析原因
   └─ 思考为什么会出现这个结果

4. 修复问题
   └─ 修改代码，重新测试

5. 验证修复
   └─ 确认问题已解决
```

---

### 3.3 Scratch → Python 映射

为后续Python学习做铺垫，可以适当提及对应关系：

| Scratch积木 | Python语法 | 说明 |
|------------|-----------|------|
| 重复 10 次 | `for i in range(10):` | 固定次数循环 |
| 重复执行 | `while True:` | 无限循环 |
| 重复执行直到 <条件> | `while not condition:` | 条件循环 |
| 如果 <条件> 那么 | `if condition:` | 单分支 |
| 如果 <条件> 那么...否则 | `if...else:` | 双分支 |
| 设定[变量]为(值) | `variable = value` | 变量赋值 |
| 将[变量]改变(值) | `variable += value` | 变量修改 |

## 四、硬件教学与调试

### 4.1 常用传感器教学

#### 红外传感器

**原理**：检测物体是否在前方

**教学话术**：
> "红外传感器就像眼睛，能看到前面有没有东西挡着。"

**应用场景**：
- 自动门
- 避障小车
- 计数器

---

#### 超声波传感器

**原理**：测量与物体的距离

**教学话术**：
> "超声波传感器就像蝙蝠，发出声波，听回声来判断距离。"

**应用场景**：
- 倒车雷达
- 智能垃圾桶
- 距离测量

---

#### 光线传感器

**原理**：检测环境光线强度

**教学话术**：
> "光线传感器就像我们的眼睛，能感觉到亮还是暗。"

**应用场景**：
- 自动路灯
- 智能窗帘
- 光控开关

---

### 4.2 常见硬件问题排查

#### 传感器不工作

| 现象 | 可能原因 | 解决方法 |
|------|---------|---------|
| 完全无反应 | 接线错误 | 检查VCC/GND/信号线 |
| 读数异常 | 电压不匹配 | 确认3.3V/5V需求 |
| 读数不稳定 | 接触不良 | 重新插拔连接线 |

#### 电机/舵机问题

| 现象 | 可能原因 | 解决方法 |
|------|---------|---------|
| 电机不转 | 电压不足 | 使用外部电源 |
| 舵机抖动 | 角度超出范围 | 限制在0-180度 |
| 舵机卡死 | 齿轮损坏 | 更换舵机 |

**⚠️ 禁忌提醒**：
- 绝对不要手动强行扭动舵机！会导致内部齿轮损坏（扫齿）

#### 快速排查流程

```
1. 检查电源
   ├─ VCC/GND是否正确连接？
   ├─ 电压是否匹配（3.3V/5V）？
   └─ 电源是否稳定？

2. 检查接线
   ├─ 信号线是否连接正确引脚？
   ├─ 是否有松动？
   └─ 是否有短路？

3. 检查代码
   ├─ 引脚号是否正确？
   └─ 是否有语法错误？

4. 检查硬件
   ├─ 设备是否损坏？
   └─ 更换设备测试
```

---

### 4.3 安全注意事项

| 注意事项 | 说明 |
|---------|------|
| 断电操作 | 接线前必须断开电源 |
| 电压确认 | 确认设备电压需求（3.3V/5V） |
| 防止短路 | 避免金属物品接触电路板 |
| 轻拿轻放 | 电子元件易损坏 |

## 五、课堂管理技巧

### 5.1 CODE 1（6-7岁）管理

**特点**：
- 注意力时间：10-15分钟
- 需要具体指令和可视化
- 容易被新事物吸引

**策略**：
| 策略 | 示例 |
|------|------|
| 分段活动 | 每10-15分钟切换活动形式 |
| 具体指令 | "把红色积木放在蓝色积木上面" |
| 视觉提示 | 用图片展示步骤 |
| 即时反馈 | "做得真棒！" |

**常用口令**：
- "小眼睛看老师"
- "小手放桌上"
- "1-2-3，坐坐好"

---

### 5.2 CODE 2（7-8岁）管理

**特点**：
- 注意力时间：15-20分钟
- 开始有合作意识
- 喜欢竞争和挑战

**策略**：
| 策略 | 示例 |
|------|------|
| 小组合作 | 2-3人一组完成任务 |
| 适度竞争 | "看哪个小组先完成" |
| 角色分配 | "你来搭建，他来编程" |
| 规则明确 | 提前说明规则和奖励 |

---

### 5.3 CODE 3（8-10岁）管理

**特点**：
- 注意力时间：20-25分钟
- 有较强的自主性
- 能进行复杂的逻辑推理

**策略**：
| 策略 | 示例 |
|------|------|
| 任务驱动 | 给出明确的挑战目标 |
| 自主探索 | 减少干预，鼓励尝试 |
| 同伴学习 | 让学生互相教 |
| 项目制 | PBL项目驱动学习 |

---

### 5.4 常见课堂问题处理

| 问题 | 处理方式 |
|------|---------|
| 学生注意力分散 | 切换活动形式，用新奇事物吸引 |
| 学生争抢材料 | 明确分配规则，强调合作 |
| 学生遇到困难放弃 | 降低难度，分步引导 |
| 学生完成太快 | 提供进阶挑战任务 |
| 学生之间冲突 | 暂停活动，引导解决 |
| 学生"看一步做一步" | 强调先完成任务分析 |

## 六、常见问题与解决方案

### 6.1 教学问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 学生不感兴趣 | 引入不够吸引 | 用故事、动画、实物引入 |
| 学生听不懂 | 概念太抽象 | 用类比、实物演示 |
| 时间不够用 | 环节时间分配不当 | 提前演练，控制每环节时间 |
| 学生差异大 | 能力水平不同 | 分层任务，同伴互助 |
| 学生不会Debug | 缺乏调试训练 | 专门讲解调试方法 |

### 6.2 编程问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 程序不运行 | 缺少启动事件 | 检查"当绿旗被点击" |
| 角色不动 | 移动步数太小 | 增加移动步数 |
| 循环不停止 | 使用了"重复执行" | 改用"重复N次"或条件循环 |
| 碰撞检测不灵 | 角色太小或太快 | 调整大小或速度 |
| 变量不更新 | 忘记修改变量 | 检查变量修改逻辑 |

### 6.3 硬件问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 传感器无反应 | 接线错误 | 检查VCC/GND/信号线 |
| 电机不转 | 电压不足 | 使用外部电源 |
| 舵机抖动 | 角度超范围 | 限制0-180度 |
| 程序上传失败 | 端口错误 | 检查COM口设置 |
